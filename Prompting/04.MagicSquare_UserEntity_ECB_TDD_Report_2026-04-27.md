```
위에서 만든 .cursorrules 뼈대의 빈 섹션을 모두 채워줘.
MagicSquare 프로젝트 기준으로 작성해.

각 섹션 작성 규칙:

code_style:
  - python_version: "3.10+"
  - style_guide: PEP8 엄격 준수
  - type_hints: 모든 함수 파라미터와 반환값에 필수
  - docstring: Google 스타일, 모든 public 메서드에 필수
  - max_line_length: 88 (Black 기준)

architecture:
  ECB 패턴 3 레이어를 각각 정의해줘:
  - boundary: 외부 입출력 담당 (UI, API, CLI)
  - control: 비즈니스 로직 담당
  - entity: 도메인 데이터 및 규칙 담당
  레이어 간 의존성 방향도 명시해줘.

tdd_rules:
  지금은 문자열 1줄인데, 하위 항목으로 세분화해줘.
  각 phase마다:
    - description: 단계 설명
    - rules: 지켜야 할 규칙 목록
    - must_not: 이 단계에서 하면 안 되는 것

testing:
  - framework: pytest
  - pattern: AAA (Arrange-Act-Assert)
  - coverage_minimum: 80%
  - fixture_scope: 규칙 정의
  - naming_convention: test_ 접두사 필수

forbidden:
  항목마다 아래 구조로 작성:
    pattern: 금지 패턴
    reason: 금지 이유
    alternative: 대신 써야 할 것
  최소 포함 항목: print(), 하드코딩 상수, except 단독 사용

file_structure:
  ECB 기준 폴더 구조를 트리 형태 주석으로 작성해줘.
  boundary/, control/, entity/, tests/ 포함.

ai_behavior:
  Cursor AI가 코드 생성 전·중·후에 반드시 따라야 할 규칙.
  최소 포함:
    - 코드 작성 전 관련 테스트 파일 확인
    - ECB 레이어 경계 위반 금지
    - 타입힌트 없는 함수 생성 금지
    - tdd_rules 위반 시 경고 출력

완성된 .cursorrules 전체 파일을 출력해줘.

################################################################################
project:
  name: MagicSquare
  language: python
  description: "ECB 아키텍처와 TDD 기반으로 매직스퀘어 로직과 UI를 개발하는 프로젝트"

################################################################################
code_style:
  python_version: "3.10+"
  style_guide: "PEP8 엄격 준수"
  type_hints: "모든 함수 파라미터와 반환값에 필수"
  docstring: "Google 스타일, 모든 public 메서드에 필수"
  max_line_length: 88

################################################################################
architecture:
  pattern: ECB
  boundary:
    role: "외부 입출력 담당 (UI, API, CLI)"
    responsibilities:
      - "사용자 입력 수집 및 출력 포맷팅"
      - "외부 요청/응답 어댑터 처리"
      - "Control 레이어 호출"
  control:
    role: "비즈니스 로직 담당"
    responsibilities:
      - "유스케이스 오케스트레이션"
      - "Entity 규칙 조합 및 실행 흐름 제어"
      - "Boundary와 Entity 사이 데이터 변환"
  entity:
    role: "도메인 데이터 및 규칙 담당"
    responsibilities:
      - "핵심 도메인 모델 정의"
      - "불변 조건 및 도메인 규칙 검증"
      - "부작용 없는 순수 계산 로직 유지"
  dependency_direction:
    allowed:
      - "boundary -> control"
      - "control -> entity"
    forbidden:
      - "boundary -> entity 직접 의존 금지"
      - "entity -> control 의존 금지"
      - "entity -> boundary 의존 금지"

################################################################################
tdd_rules:
  red_phase:
    description: "실패하는 테스트를 먼저 작성해 요구사항을 명확히 한다."
    rules:
      - "새 기능/버그 수정 전 테스트 케이스를 먼저 추가한다."
      - "테스트 실행 결과가 실패(Fail)임을 확인한다."
      - "실패 원인이 요구사항 미구현임을 확인한다."
    must_not:
      - "테스트 없이 프로덕션 코드를 먼저 작성하지 않는다."
      - "여러 기능을 한 번에 검증하는 과도한 통합 테스트를 먼저 작성하지 않는다."
  green_phase:
    description: "추가한 테스트를 통과시키는 최소한의 구현만 수행한다."
    rules:
      - "테스트 통과를 위한 최소 코드만 작성한다."
      - "기존 테스트 전체를 실행해 회귀가 없는지 확인한다."
      - "도메인 규칙은 Entity에, 흐름 제어는 Control에 배치한다."
    must_not:
      - "구조 개선 목적의 리팩터링을 수행하지 않는다."
      - "가독성 개선을 이유로 범위를 확장하지 않는다."
      - "테스트 통과와 무관한 최적화를 하지 않는다."
  refactor_phase:
    description: "기능 동작을 바꾸지 않고 구조를 개선한다."
    rules:
      - "동작 동일성을 유지하면서 중복 제거와 명확한 네이밍을 적용한다."
      - "리팩터링 전후 테스트를 모두 통과시킨다."
      - "커버리지를 최소 기준 이상으로 유지한다."
    must_not:
      - "새 기능을 추가하거나 요구사항을 변경하지 않는다."
      - "테스트를 삭제해 통과율을 인위적으로 높이지 않는다."
      - "레이어 경계를 깨는 의존성 변경을 하지 않는다."

################################################################################
testing:
  framework: pytest
  pattern: AAA
  coverage_minimum: "80%"
  fixture_scope:
    default: "function"
    integration: "module"
    session_allowed_when: "외부 리소스 초기화 비용이 크고 테스트 간 상태 공유가 안전한 경우에만 허용"
  naming_convention: "모든 테스트 파일/함수에 test_ 접두사 필수"

################################################################################
forbidden:
  - pattern: "print()"
    reason: "디버그 출력이 실행 경로에 남아 로그 품질을 저하시킴"
    alternative: "logging 모듈의 구조화 로그(logger.debug/info/warning/error) 사용"
  - pattern: "하드코딩 상수"
    reason: "매직 넘버/문자열이 유지보수성과 변경 대응력을 떨어뜨림"
    alternative: "entity 상수, Enum, 설정 파일(config) 또는 환경변수로 분리"
  - pattern: "except:"
    reason: "예외 원인 은닉으로 디버깅 및 장애 분석이 어려워짐"
    alternative: "except SpecificError as e 형태로 명시적 예외 처리 후 로깅/재전파"

################################################################################
file_structure:
  root: "src/"
  # src/
  # ├─ boundary/    # UI, API, CLI, 외부 인터페이스 어댑터
  # ├─ control/     # 유스케이스, 애플리케이션 서비스, 흐름 제어
  # ├─ entity/      # 도메인 모델, 규칙, 값 객체
  # └─ tests/       # pytest 테스트 코드 (unit/integration)
  folders:
    - "boundary/"
    - "control/"
    - "entity/"
    - "tests/"

################################################################################
ai_behavior:
  before_code_generation:
    - "관련 테스트 파일(기존 unit/integration 테스트)을 먼저 확인한다."
    - "변경 대상이 속한 ECB 레이어를 식별하고 경계 위반 가능성을 점검한다."
    - "추가/수정 함수에 필요한 타입힌트와 docstring 요구사항을 사전 체크한다."
  during_code_generation:
    - "ECB 레이어 경계를 위반하는 호출/의존성을 만들지 않는다."
    - "타입힌트 없는 함수를 생성하지 않는다."
    - "Green 단계에서는 테스트 통과 최소 구현만 작성하고 리팩터링을 수행하지 않는다."
  after_code_generation:
    - "pytest를 실행해 추가 테스트 및 전체 회귀 테스트를 확인한다."
    - "tdd_rules 위반이 감지되면 경고를 출력하고 위반 항목을 명시한다."
    - "리팩터링 이후 커버리지가 coverage_minimum 미만으로 떨어지지 않았는지 확인한다."
```

--- 반드시 포함할 섹션 ---

1. project          → 프로젝트명, 설명, 방법론
2. code_style       → PEP8 엄격 준수, 타입힌트 필수(Python 3.10+), Google 스타일 docstring
3. architecture     → ECB 패턴 (boundary / control / entity 레이어별 역할 정의)
4. tdd_rules        → red_phase / green_phase / refactor_phase 각 단계별 규칙
5. testing          → pytest AAA 패턴(Arrange-Act-Assert), fixture 규칙, 커버리지 기준
6. forbidden        → print() 디버깅 금지, 그 외 금지 패턴 목록
7. file_structure   → 레이어별 폴더 구조 예시
8. ai_behavior      → 코드 생성 시 Cursor AI가 반드시 따라야 할 행동 규칙

--- 형식 규칙 ---
- YAML 형식, 들여쓰기 2 스페이스
- 각 섹션 앞에 80자 # 구분선 + 섹션 제목 주석
- 키는 영어, 설명은 한국어로 작성
- tdd_rules 하위에 red_phase / green_phase / refactor_phase 동일 깊이로 작성
- forbidden 항목마다 reason(이유)과 alternative(대안)을 함께 명시
