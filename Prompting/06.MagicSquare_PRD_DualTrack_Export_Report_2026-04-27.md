## Dual-Track UI + Logic TDD — 방향은 타당합니다

슬라이드에 나온 구조( **UX Contract로 Boundary RED**, **Logic Rule로 Domain RED**, **GREEN/REFACTOR는 함께**, **UI 테스트 ↔ 로직 테스트 완전 독립** )는 지금 PRD의 Boundary / Domain 분리·계약 기반 TDD와 같은 축입니다.  
추가로 **6.2의 UX Contract / Logic Rule 언어 + 시나리오 매핑 표**는 요구사항을 “테스트로 바꿀 수 있는 판단(Decision)”으로만 내리게 해 주어서, To-Do를 남발하지 않게 만드는 데도 잘 맞습니다.

---

## “with MLOps” — 여기서 한 번만 정리하면 좋습니다

**MLOps**는 보통 *학습 데이터 → 실험·버전 → 평가 → 배포 → 모니터링(드리프트 등)*까지 묶는 말이라, **4×4 마방진의 핵심 도메인(합 34, 빈칸 2개, 두 조합 시도)** 은 **결정론·규칙 기반**이어서 “도메인 진리”를 ML이 대체하기는 부적절합니다. 그래서 조합은 이렇게 나누는 걸 권합니다.

| 레이어 | 역할 | TDD |
|--------|------|-----|
| **UI** | 보임/안 보임, 활성/비활성, 오류 문구·스타일 | UX Contract 테스트 |
| **Domain** | 허용/거부, 합·빈칸·해 탐색 | Logic Rule 테스트 (지금 PRD의 FR) |
| **ML(선택)** | *규칙을 대체하지 않음* — 힌트, 난이도 추정, 입력 패턴 이상 탐지 등 **보조** | 별도 계약 + 오프라인 평가지표 |

즉, **Dual-Track TDD는 그대로 두고**, MLOps는 **“세 번째 궤도”**처럼 **데이터/모델 파이프라인과 운영**에만 두는 그림이 깔끔합니다.

---

## MLOps를 붙일 만한 현실적인 포인트 (예시)

- **모델이 틀려도 제품이 깨지지 않게**: UI/API는 여전히 Domain 결과만 신뢰하고, ML 출력은 “추천/설명” 정도로만 표시.
- **계약 분리**: ML 입출력은 `features` 스키마·버전·지연시간 SLI처럼 **별도 문서/테스트**(스키마 검증, 회귀용 고정 픽스처).
- **파이프라인**: 학습/재학습은 CI와 분리하거나, `train` → `eval` 게이트(임계값) → 아티팩트 버전 태깅 정도만 먼저 잡아도 MLOps “맛”이 납니다.

슬라이드 경고대로, **“동사로 끝나는 To-Do”가 아니라 “판단이 있는 항목”만** 테스트로 내리고, ML 쪽도 **“언제 허용/거부/폐기할지”** 같은 **정책(Decision)** 을 명시해야 TDD·MLOps가 같이 안 꼬입니다.

---

## 짧은 결론

- **Dual-Track UI + Logic TDD**: 지금 프로젝트 철학과 잘 맞고, 계속 밀어도 됩니다.  
- **MLOps**: **핵심 로직을 ML로 바꾸기보다**, **보조 기능 + 실험/배포/모니터링 규율**으로 붙이면 의미가 분명합니다.  
- 먼저 **“ML이 사용자에게 무엇을 보여줄지(한 문장)”** 와 **“Domain 판정과 어떻게 독립할지”** 만 정하면, PRD 8장 옆에 **Track C (ML / Ops)** 한 절을 추가하는 식으로 문서도 정리하기 쉽습니다.

원하시면, 지금 PRD의 FR/TC 틀에 맞춰 **Track C에 넣을 문단 목차(계약·지표·게이트만)** 를 짧게 초안으로 잡아 드릴 수 있습니다.

## 1. Executive Summary (요약 — 재정리)

- 목표는 “알고리즘 난이도”가 아니라 **고정 입출력 계약·불변조건을 TDD로 고정**하는 것.
- **Track A**: 입력 검증·출력 포맷·오류 코드·Domain 예외 매핑(**Boundary / UX·I/O 계약**).  
- **Track B**: 빈칸·누락 수·마방진 판정·두 조합 해(**순수 Domain**).  
- **Track C**(선택): **정답 규칙을 ML이 대체하지 않고**, 데이터/모델 **버전·평가·배포·모니터링**을 체계화(MLOps).
- 기존 **NFR**, **결정론**, **레이어 의존성**(`boundary → control → entity`)은 그대로 유효.

---

## 2. Problem Statement & Target Users

- **문제 정의**: “마방진을 생성”이 아니라 “4×4가 **검증 가능한 계약** 아래 완성·판정된다.”
- **사용자**: TDD 학습자, 리뷰어, 실행 환경(콘솔 / `pytest` 등) — **동일**.

---

## 3. Scope (범위 — Dual-Track + MLOps로 재표기)

| 구분 | 내용 |
|------|------|
| **In-Scope (Track A+B)** | 기존 4.1과 동일: Boundary 검증, row-major 빈칸, 누락 2개, 합 34 판정, `int[6]` 반환, 두 시도 규칙. |
| **In-Scope (Track C, 선택)** | ML이 있다면: **훈련/실험 아티팩트 버전**, **오프라인 지표**, **서빙 입출력 스키마**, **롤백/모니터링**만 — **BR/FR의 진리는 Domain**. |
| **Out-of-Scope (기존)** | “풀 UI”는 원문 4.2에 따름. *실제로 웹 UI를 붙이면* 이 항목은 **범위 변경**이 필요(Track A를 화면 단에서도 같은 계약으로 검증). |
| **Out-of-Scope (MLOps)** | **N×N 일반화**는 기존과 같이 확장만; **도메인 규칙 전체를 학습에 맡기기**는 Out-of-Scope 권장. |

---

## 4. 언어 사전: UX Contract vs Logic Rule (6.2식 정렬)

- **UX Contract(Track A)**: *보인다/안 보인다*, *가능/불가*, *활성/비활성*, *오류 UI·메시지 노출* — **실제론 I/O·경계**로 구현(콘솔·API·React 공통 “계층 계약”).  
- **Logic Rule(Track B)**: *허용/거부*, *계산/중단*, *반환/차단* — **FR-02~05, BR-01~12**.  
- **MLOps(Track C)**: *승인/거부(모델 배포)*, *지표 하락 시 롤백* 등 — **FR과 별도**로 “운영 정책”만.

**3단 매핑 예시(기존 TC에 대응)**

| 시나리오 | UX Contract (Track A) | Logic Rule (Track B) |
|----------|------------------------|----------------------|
| 합이 34가 아님(검증 단계) | (해당 시) 오류/실패 **표시** | 합 검증 **거부** / `isMagic` **false** |
| 빈칸 수 ≠ 2 | “빈칸 오류” **표시** (동일 code) | 탐색/해 **중단** → Boundary가 `E_UI_BLANK_COUNT` |
| 올바른 입력으로 완성 | “완료” **표시** | `int[6]` **반환** |
| 잘못된 정수(경계) | 필드/입력 **강조** 등 | 입력 **차단** → `E_UI_VALUE_RANGE` … |

(원문 FR-01~05의 AC·오류코드가 이 표의 “단일 출처”입니다.)

## 5. Functional Requirements — 트랙별 매핑

| Track | ID | 요약 |
|--------|----|------|
| **A (Boundary / “UI” TDD)** | **FR-01** | 입력 검증, `E_UI_*`, `{code,message}`, Domain으로만 통과 |
| | *(출력/매핑)* | `E_UI_OUTPUT_FORMAT`, `E_DOMAIN_NO_SOLUTION` 매핑, 길이·1-index **계약** |
| **B (Domain / Logic TDD)** | **FR-02~05** | 빈칸, 누락, `isMagic`, `solve` 두 시도, `SolveFailure` → Track A가 매핑 |
| **C (MLOps, 선택)** | *신규 명명 예: ML-01~* | 데이터/모델 **스키마·버전**, **오프라인 eval 게이트**, **서빙 SLA**, **드리프트 알람** — **FR-01~05 결과를 덮지 않음** |

**Business Rules(BR-01~12)**: 전부 **Track B 진리** + **Track A 출력·불변성(좌표·배열·결정론)** 반영. 변경 없이 **“B 우선, A는 계약”**이면 됩니다.

| 트랙 | RED의 재료 | GREEN/REFACTOR | 금지 |
|------|------------|----------------|------|

| **Track A — UX / Boundary** | **내용은 있으나 이름이 애매함.** FR-01, 8.1, 오류 코드·포맷·매핑은 “경계 계약”으로 충분한데, 제목만 “UI”이고 **4.2에서는 GUI를 Out-of-Scope**로 두어 **슬라이드의 “UI = React”** 와 **문서의 “콘솔/pytest”** 가 충돌합니다. |
| **Track C — MLOps** | **없음.** NFR·아키텍처·추적표에 **데이터/모델/배포/모니터링** 한 줄도 없어, “with MLOps” 정리와는 **아직 미정렬**입니다. |

---

### 2. 문서 내부의 긴장(모순은 아니지만 정리 필요)

- **4.2** “UI 화면 개발 Out-of-Scope” vs **8.1** “Boundary(**UI**) TDD”  
  - 의도는 “**I/O·경계 = UI에 준하는 계약**”으로 읽을 수 있지만, 문서만 보면 **실제 UI를 할지 말지**가 불명확합니다.  
- **10절** Boundary “**Entity 직접 의존 금지**”는 ECB 관점에선 맞는데, **Control이 Entity를 쓰는** 전형적 구조와 같이 읽으면 혼동 여지가 있습니다(“Boundary가 Entity를 import하면 안 됨” 수준으로 **한 문장 보강**이 있으면 좋음).
- **FR 섹션(5)** 은 **FR-01 = Track A**, **FR-02~05 = Track B**로 사실상 나뉘는데, **절 제목에 Track 라벨이 없어** 슬라이드의 “UX vs Logic 3단 매핑”과 **1:1로 붙이기는 어렵습니다.**

---

### 3. 잘 된 점 (유지 권장)

- **입력/출력·오류 코드·AC**가 촘촘해 **SSOT**에 가깝고, **8.3 병렬 사이클**과 **금지 규칙**이 TDD 교육 목적과 잘 맞습니다.
- **9절** TC / TD / PI로 **검증 층**이 나뉘어 있고, **12절** 추적표로 **BR–FR–TC–컴포넌트**가 잡혀 있습니다.
- **D-01~D-04**로 애매한 설계 결정이 문서에 고정되어 있습니다.

---

### 4. 정리한 컨셉 기준 “빈 칸”

1. **UX Contract vs Logic Rule**  
   - 슬라이드 6.2의 **동사/상태 사전**과, **시나리오 → UX / Logic** 한 표가 PRD에 **없음** → Track A/B를 **같은 시나리오 ID**로 엮기 어렵습니다.  
2. **“판단 있는 것만 테스트로”**  
   - 원칙이 문서에 **명시되지 않음** (9절은 있으나 **원칙 문장**이 없음).  
3. **MLOps**  
   - **Out-of-Scope(선택)**、**Track C**、**Domain 우선 / 모델 보조** 같은 **한 단락**이 없으면, 이전에 합의한 **3트랙 PRD**와 **불일치**로 남습니다.  
4. **추적표(12)**  
   - **Track (A/B/C)** 열이 없어, 나중에 ML-01 같은 요구가 생기면 **같은 표에 못 붙임**.

### 5. 총평

- **Dual-Track의 실질(경계 vs 도메인, 독립 테스트, 병렬 RED/GREEN)** 은 **8·5·6·7절에 이미 반영**되어 있습니다.  
- **“UI + Logic with MLOps”로 말한 컨셉**과 맞추려면, 문서는 아직 **(1) GUI 범위와 “UI” 용어 정리**, **(2) MLOps 트랙(또는 명시적 비적용)**, **(3) UX/Logic 언어·시나리오 매핑 1절** 정도를 **보완**하는 단계로 보는 것이 정확합니다.
