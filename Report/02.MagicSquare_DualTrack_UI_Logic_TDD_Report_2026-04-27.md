# Magic Square (4x4) — Dual-Track UI + Logic TDD 설계 보고서

## 프로젝트 개요
- 목적: 알고리즘 난이도보다 **레이어 분리 + 계약 기반 테스트 + 리팩토링** 훈련
- 범위: 구현 코드 제외, 설계/계약/테스트/통합 계획만 정의
- UI 정의: 실제 화면이 아닌 입력/출력 **Boundary**
- Data 정의: DB 제외, 저장/로드 인터페이스(메모리/파일 교체 가능) 수준

## 입력/출력 고정 계약
- 입력: `4x4 int[][]` (0은 빈칸)
- 입력 규칙:
  - 빈칸(0)은 정확히 2개
  - 값 범위는 `0` 또는 `1~16`
  - `0` 제외 중복 금지
- 출력: `int[6]`
  - 좌표는 1-index
  - 형식: `[r1,c1,n1,r2,c2,n2]`
  - `n1,n2`는 누락 숫자 2개
  - (작은수→첫빈칸, 큰수→둘째빈칸) 조합이 마방진이면 그 순서, 아니면 반대 순서

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 구성요소 | 분류 | 책임(SRP) | 입력 | 출력 |
|---|---|---|---|---|
| MagicSquareBoard | Entity | 4x4 상태 보관, 셀 조회, 빈칸 좌표 제공 | int[4][4] | 셀 값/좌표 |
| CellPosition | Value Object | 1-index 좌표 값 표현 및 범위 검증 | (row, col) | 유효 좌표 |
| MissingNumbers | Value Object | 0 제외 누락 숫자 2개 계산 결과 보관 | 보드 값 집합 | (small, large) |
| MagicSquareValidator | Domain Service | 행/열/대각선 합 = Magic Constant 검증 | 완성 보드 | isMagic(boolean) |
| TwoAssignmentSolver | Domain Service | small/large 두 조합 시도 후 정답 순서 결정 | 보드+빈칸+누락숫자 | [r1,c1,n1,r2,c2,n2] |

## 1.2 도메인 불변조건(Invariants)

| ID | 불변조건 | 검증 방법 | 실패 시 |
|---|---|---|---|
| INV-01 | 보드 크기는 항상 4x4 | 행 수=4, 각 행 길이=4 | DomainValidationError |
| INV-02 | 0 값은 정확히 2개 | 0 카운트==2 | DomainValidationError |
| INV-03 | 각 셀은 0 또는 1~16 | 모든 셀 범위 검사 | DomainValidationError |
| INV-04 | 0 제외 숫자 중복 없음 | Set 크기와 원소 수 비교 | DomainValidationError |
| INV-05 | 완성된 마방진의 각 행/열/대각선 합은 34 | 합 10개 전부 34 비교 | SolveFailure |

## 1.3 핵심 유스케이스(도메인 관점)

| UC | 이름 | 사전조건 | 절차 | 완료조건 |
|---|---|---|---|---|
| UC-D1 | 빈칸 찾기 | 입력 보드 유효 | 0인 셀 2개를 행우선 순서로 수집 | 두 좌표 확보 |
| UC-D2 | 누락 숫자 찾기 | 0 제외 중복 없음 | 1~16 전체에서 현재 숫자 제거 | 작은수/큰수 확보 |
| UC-D3 | 마방진 판정 | 보드 완성(0 없음) | 행/열/대각선 10개 합 계산 | True/False |
| UC-D4 | 두 조합 시도 | 빈칸2+누락2 확보 | 케이스A(small→빈칸1, large→빈칸2) 후 실패 시 반대 | 출력 int[6] 결정 |

## 1.4 Domain API(내부 계약)

| 메서드 시그니처 수준 | 입력 | 출력 | 실패조건 |
|---|---|---|---|
| MagicSquareBoard.from(matrix: int[4][4]) | 4x4 int[][] | MagicSquareBoard | INV-01~04 위반 |
| board.findBlankPositions(): [CellPosition, CellPosition] | 보드 | 빈칸 2개(행우선) | 빈칸 개수!=2 |
| board.findMissingNumbers(): MissingNumbers | 보드 | (small, large) | 중복/범위 위반 |
| validator.isMagic(board): boolean | 완성 보드 | true/false | 완성되지 않은 보드 입력 |
| solver.solve(board): int[6] | 유효 보드 | [r1,c1,n1,r2,c2,n2] | 두 조합 모두 마방진 실패 |

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

| 단계 | 행동 | 검증 | 결과 |
|---|---|---|---|
| S1 | 4x4 행렬 입력 | 스키마/범위/중복/빈칸2 검사 | 유효 시 다음 단계 |
| S2 | Solve 요청 | Domain 서비스 호출 | 성공 시 int[6] 수신 |
| S3 | 결과 출력 | 출력 배열 길이/좌표범위 검증 | 표준 포맷 반환 |

## 2.2 UI 계약(외부 계약)

| Contract | Schema | 필드 규칙 | 검증 포인트 |
|---|---|---|---|
| Input schema | `{ matrix: int[4][4] }` | 0=정확히2, 값=0\|1..16, 0 제외 unique | 요청 수신 즉시 |
| Output schema | `{ result: int[6] }` | 1-index, `[r1,c1,n1,r2,c2,n2]` | Domain 응답 직후 |
| Error schema | `{ code, message, details? }` | code 고정 집합, message 표준 문구 | 모든 실패 경로 |

## 2.4 UX/출력 규칙

| Code | 메시지(고정) | 노출 조건 | 재현 테스트 |
|---|---|---|---|
| E_UI_MATRIX_SIZE | Matrix must be 4x4. | 행/열 크기 위반 | T-U-01 |
| E_UI_BLANK_COUNT | Matrix must contain exactly two zeros. | 0 개수 != 2 | T-U-02 |
| E_UI_VALUE_RANGE | Values must be 0 or between 1 and 16. | 값 범위 위반 | T-U-03 |
| E_UI_DUPLICATE | Non-zero values must be unique. | 0 제외 중복 | T-U-04 |
| E_UI_OUTPUT_FORMAT | Result must be [r1,c1,n1,r2,c2,n2] with 1-index coordinates. | 출력 계약 위반 | T-U-06 |

에러 메시지 표준 규칙:
- 문구는 완전 일치 문자열 비교 테스트로 고정
- 임의 문구 변경 금지
- 변경 시 계약 버전 증가 + 관련 테스트 전부 업데이트 필수

# 3) Data Layer 설계 (Data Layer)

## 3.1 목적 정의
- 저장/로드 경계 분리 훈련
- 재현 가능한 입력/결과 보관
- DB 연동 제외, 인터페이스 중심 설계

## 3.2 인터페이스 계약

| 메서드 | 입력 | 출력 | 실패조건 |
|---|---|---|---|
| MatrixRepository.saveInput(matrix, runId) | 4x4 matrix + runId | void | 형식 불일치/쓰기 실패 |
| MatrixRepository.loadInput(runId) | runId | 4x4 matrix | 없음/형식 오류 |
| MatrixRepository.saveResult(runId, result) | runId + int[6] | void | 길이 오류/쓰기 실패 |
| MatrixRepository.loadResult(runId) | runId | int[6] | 없음/형식 오류 |

## 3.3 구현 옵션 비교(메모리/파일)

| 옵션 | 장점 | 단점 | 테스트 난이도 | 추천 |
|---|---|---|---|---|
| A: InMemory | 빠름, 결정적, 테스트 격리 쉬움 | 프로세스 종료 시 소실 | 낮음 | 기본 |
| B: File(JSON/CSV) | 재시작 후 재현 가능 | I/O 실패/형식 오류 처리 필요 | 중간 | 확장 |

추천안: **옵션 A(InMemory)**  
이유:
- TDD 반복 속도가 가장 빠름
- 실패 원인 추적이 도메인/계약에 집중됨
- 학습 목적(경계 분리 + 테스트)과 정합

## 3.4 Data 레이어 테스트

| 테스트 ID | 대상 | 시나리오 | 기대결과 |
|---|---|---|---|
| T-DA-01 | save/load input | 저장 후 즉시 로드 | 4x4 동일성 유지 |
| T-DA-02 | save/load result | 결과 저장 후 로드 | int[6] 동일성 유지 |
| T-DA-03 | load missing id | 없는 runId 조회 | DataNotFoundError |
| T-DA-04 | file parse | 손상된 JSON/CSV | DataFormatError |
| T-DA-05 | invariant | 로드 데이터가 4x4 아님 | DataInvariantError |
