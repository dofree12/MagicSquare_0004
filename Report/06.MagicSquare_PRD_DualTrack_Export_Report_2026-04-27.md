## 4) PRD v0.2 핵심 구성 요약

### 4.1 프레임워크 (§3)

- **Track A — UX / I/O Contract (Boundary)**: 입력 검증·오류 코드·`int[6]`·1-index 등 **외부 계약**; GUI 없을 때는 콘솔·`pytest`·API가 구현체.  
- **Track B — Logic Rule (Domain)**: BR-01~12, FR-02~05; **SSOT(정답·규칙)**.  
- **Track C — MLOps (선택)**: 아티팩트 버전·eval 게이트·서빙 스키마·롤백·SLI; **FR 비즈니스 판정을 ML이 대체하지 않음**.  
- **핵심 문장**: *UI는 UX·I/O 계약으로 RED, 로직은 규칙으로 RED — GREEN·REFACTOR는 함께*; Track A/B 테스트 **상호 독립**.

### 4.2 문서 메타·범위 (§5)

- 상단 표: 버전 0.2, Draft, 목적 문구.  
- **In-Scope**: 기존 Boundary/Domain 요구 + **(선택) MLOps 절차** (도메인 규칙 대체 아님).  
- **Out-of-Scope**: **v0.2 기준 풀 GUI 제품**은 제외; 향후 GUI 시 **동일 FR/AC·오류 코드**로 Track A 검증 권장. **도메인 정답을 학습 모델이 대체**하는 것은 Out.

### 4.3 UX / Logic 언어·시나리오 매핑 (§6)

- **§6.1** 언어 사전: UX·I/O vs Logic Rule 동사·상태.  
- **§6.2** 대표 시나리오 3단 매핑 표.  
- **§6.3** **판단(Decision)이 있는 요구만** Track A/B 테스트로 변환 (To-Do 남발 방지).

### 4.4 고정 계약 (기존 + Track 라벨)

- 입력: 4×4 `int[][]`, 0은 빈칸, 빈칸 2개, 값 범위(0 또는 1~16), 0 제외 중복 금지, row-major 첫 빈칸 정의.  
- 출력: `int[6] = [r1,c1,n1,r2,c2,n2]`, 1-index, 두 조합 시도 규칙 고정.  
- **Track 매핑 (§7)**: FR-01 → **A**; FR-02~05 → **B** (FR-05 출력은 A와 **계약 경계**).

### 4.5 기능·규칙

- **FR-01~05**: 본문 동일, 각 절에 Track 라벨 추가.  
- **BR-01~BR-12**: §8, Domain(SSOT) 유지.

### 4.6 비기능 (§9)

- 기존 NFR-01~05 + **NFR-06 (선택) MLOps 품질 게이트**; NFR-02/04에 Track C 재현성·SLI 연계 문장.

### 4.7 TDD 전략 (§10)

- **§10.1~10.2** Track A/B (기존 8.1~8.2 대응, 제목에 UX/I/O 명시).  
- **§10.3** 병렬 사이클: **I/O RED & Logic RED** → I/O GREEN & Logic GREEN → REFACTOR.  
- **§10.4 Track C**: 선택, Domain과 **비동기** 가능, D-05 준수.

### 4.8 테스트 계획 (§11)

- TC-01~08에 **A/B** 권장 태그; **§6.3** 회귀 정책 참조. TD·PI 유지.

### 4.9 아키텍처 (§12)

- Boundary **금지** 사항을 **“Domain 규칙 중복·Entity 직접 import”** 로 명확화; **(선택) MLOps/Model Serving** 레이어, **entity → ml 역참조 금지**.

## 6) 의사결정 D-01 ~ D-05 (PRD §13)

| ID | 요지 |
|----|------|
| D-01 | `SolveFailure` + `E_DOMAIN_NO_SOLUTION` 매핑 |
| D-02 | 외부 계약은 항상 1-index |
| D-03 | 첫 빈칸 = row-major 고정 |
| D-04 | 입력 원본 불변(필요 시 복사) |
| D-05 | **ML·추론 vs Domain**: 충돌 시 **Track B + Track A 오류/출력 계약**이 **항상 우선**; UI/API에 보이는 “해”는 Domain이 계약에 따라 산출 |

## 7) Traceability 반영 상태 (PRD §14)

PRD **Section 14**에 다음 연쇄가 **Track 열** 포함 표로 고정되었다.

`Track -> Concept/Invariant -> Business Rule -> Feature(FR) -> Acceptance Criteria -> Test Case -> Component`

포함 항목(Track 열: 대부분 A, B, 또는 A·B):  
- 4×4 고정 크기, 값 범위, 빈칸 2개, 0 제외 중복 금지, row-major 첫 빈칸, 누락 오름차순, 합 34, 두 조합, 출력 `int[6]`·1-index, 결정론, 입력 불변성.  
- **Track C** 전용 AC가 생기면 **같은 표에 `C` 행**으로 별도 추가(현재 v0.2는 선택 트랙).
