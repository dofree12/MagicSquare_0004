# Magic Square 4×4 완성 시스템

부분적으로 비어 있는 4×4 마방진(빈칸 `0` 두 칸)을 **도메인 규칙**에 따라 완성하거나, 유효하지 않으면 **표준 오류**로 거부하는 시스템입니다. 본 프로젝트는 Dual-Track TDD(경계·도메인 분리)와 ECB(Entity / Control / Boundary)를 전제로 합니다.

## Epic (Report 보조 참조)

**Invariant(불변조건) 기반 사고 훈련 시스템 구축** — 4×4 마방진을 통해 계약 기반 TDD·레이어 분리·추적 가능한 테스트를 연습하는 것이 목표입니다.

**Report에 제시된 성공 지표(참고):** Domain 테스트 커버리지, 입력 검증 계약 테스트 통과, Invariant→Test 추적 가능.  
**PRD 기준 정량 목표:** 아래 [검증 기준 (PRD)](#검증-기준-prd)의 **NFR-01**을 따릅니다.

## 사용자 여정 (User Journey)

1. **문제 인식:** “알고리즘 하나”가 아니라 **불변조건·계약**으로 문제를 본다.  
2. **계약 정의:** 입력/출력 스키마, 예외·오류 코드 정책 (Track A + PRD FR-01).  
3. **Domain 분리:** 빈칸 탐색, 누락 숫자, 마방진 판정, 두 조합 해 탐색 (Track B, FR-02~05).  
4. **Dual-Track:** I/O RED와 Logic RED를 병렬로 진행 → GREEN → REFACTOR (PRD §10.3).  
5. **회귀 보호:** TC·AC·PI 기반 엣지·실패 케이스 유지 (PRD §11).

## User Story 요약 (PRD FR과 대응)

| 스토리 | PRD | 한 줄 |
|--------|-----|--------|
| 입력 검증 | FR-01 | 4×4, 0 두 칸, `1..16` 또는 `0`, 0 제외 중복 금지 → 위반 시 표준 오류 |
| 빈칸 탐색 | FR-02 | row-major로 빈칸 두 좌표 (출력 계약은 1-index) |
| 누락 숫자 | FR-03 | `1..16`에서 누락 2개 → `(small, large)` |
| 마방진 판정 | FR-04 | 10개 합이 모두 34일 때만 참 (0 없는 보드) |
| 해 찾기 | FR-05 | 시도1: small→첫 빈칸, large→둘째 빈칸; 실패 시 시도2 역배치; 성공 시 `int[6]`; 둘 다 실패 시 `SolveFailure` → Boundary `E_DOMAIN_NO_SOLUTION` |

## 도메인 규칙 (요약, 상세는 PRD BR-01~12)

- 보드는 **4×4**이고, 셀 값은 **`0` 또는 1~16**만 허용한다.  
- **`0`은 정확히 2칸**(빈칸)이다.  
- **0을 제외한 중복**은 허용하지 않는다.  
- 완성 시 **행·열·대각선 10개 합**이 모두 **34**이어야 한다 (PRD FR-04).  
- 유효하지 않은 **외부 입력**은 Track A에서 거부; Domain 전제 위반은 **Domain 예외**로 처리 (PRD §7, §12).  
- **도메인 SSOT는 Track B**; ML/추론은 PRD Track C(선택)이며 **FR을 대체하지 않는다** (D-05).

---

## 아키텍처 (ECB, PRD §12)

- **Entity (Domain, Track B):** 순수 로직 — 빈칸·누락 숫자·`isMagic`·`solve` 등.  
- **Boundary (Track A):** 입출력 계약, 오류 코드·메시지, **Domain을 직접 호출하지 않음** (Control 경유).  
- **Control:** 검증 통과 후 Domain 호출·예외 매핑·흐름 조율.  
- **의존성 방향:** `boundary → control → entity` (NFR-05).

PRD §14의 추적 표에 나온 컴포넌트명(예: `BoundaryInputValidator`, `BlankPositionFinder`, `TwoAssignmentSolver`, `ResultFormatter`, `SolverOrchestrator`)을 구현 시 참고한다.

## 검증 기준 (PRD)

### 기능 요구 (FR) 승인 기준

- **FR-01 (Track A):** AC-FR01-01~05 — `E_UI_MATRIX_SIZE`, `E_UI_BLANK_COUNT`, `E_UI_VALUE_RANGE`, `E_UI_DUPLICATE`, 유효 시 Domain으로 진행.  
- **FR-02~05 (Track B):** AC-FR02-01~02, AC-FR03-01~03, AC-FR04-01~02, AC-FR05-01~05 및 BR-01~12.

### 시나리오 테스트 ID (PRD §11)

| ID | 내용 | Track |
|----|------|--------|
| TC-01 | 정상 성공(시도 1 성공) | A+B |
| TC-02 | 시도 1 실패·시도 2 성공 | A+B |
| TC-03 | 해 없음(두 시도 실패) | A+B |
| TC-04 | 입력 크기 오류 | A |
| TC-05 | 값 범위 오류 | A |
| TC-06 | 0 개수 오류 | A |
| TC-07 | 0 제외 중복 오류 | A |
| TC-08 | 출력 형식(경계 방어) | A |

대표 데이터는 PRD **TD-01~TD-07**을 사용한다.

### Property / Invariant (PI-01~05)

유효 입력에 대해: 반환 길이 6, 좌표 1..4, 누락 수와 일치, 완성 보드 `isMagic`, **입력 원본 불변** (NFR-03, AC-FR05-05).

### 비기능 (NFR)

- **NFR-01:** Domain ≥ **95%**, Boundary ≥ **85%** (라인 커버리지).  
- **NFR-02:** 동일 입력 → 동일 출력/오류.  
- **NFR-03:** 입력 `int[][]` **부작용 금지** (깊은 동등).  
- **NFR-04:** 단일 4×4 요청, 1000회 평균 **50ms** 이내(개발 머신 기준).  
- **NFR-05:** ECB 의존 방향 준수.
