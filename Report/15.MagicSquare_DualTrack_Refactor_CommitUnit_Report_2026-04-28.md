## 1. 작업 목적

- Dual-Track(UI/Logic) 분리를 유지한 상태에서, 외부 계약을 바꾸지 않는 소규모 리팩토링 1단위를 수행한다.
- 테스트 안전망을 유지하며 내부 중복/결합도를 낮춘다.

---

## 2. 이번 리팩토링 목표 (선택 ID)

- `R-L1`: Domain 좌표 표현을 내부 Value Object로 정리
- `R-U2`: Boundary 예외 코드/메시지 raise 경로 단일화

선택 사유:
- 변경 범위가 작고 위험이 낮음
- 외부 계약(입력/출력/예외)을 유지한 채 내부 구조 개선 가능

---

## 3. 변경 범위 요약

### 3.1 Logic Track (`src/magicsquare/domain.py`)

- `Coordinate`(불변 dataclass) 도입: 내부 좌표 표현 정리
- `_validate_expected_blank_count()` 추출: blank 개수 검증 책임 분리
- `find_blank_coords()`는 기존 반환 타입(`tuple[tuple[int,int], tuple[int,int]]`) 유지

### 3.2 UI Track (`src/magicsquare/boundary.py`)

- `_raise_matrix_size_error()` 추출: 동일 예외 raise 로직 중복 제거
- `MSG_E_UI_MATRIX_SIZE`를 `MATRIX_SIZE` 기반 메시지로 구성
- `validate()` 공개 계약(예외 타입/코드/메시지) 유지

## 4. 보호 테스트(안전망) 보강

리팩토링 직접 영향 범위를 고정하기 위해 최소 안전망 테스트를 추가했다.

## 7. 위험 요소 및 롤백 포인트

위험 요소:
- `MSG_E_UI_MATRIX_SIZE`가 `MATRIX_SIZE`에 연동되어 상수값 변경 시 메시지 문구도 함께 변함
- `Coordinate`는 내부 구현 타입이므로 외부 API 영향은 없으나 내부 의존 증가 가능성 존재

롤백 포인트:
- `domain.py`에서 `Coordinate` 및 `_validate_expected_blank_count()` 제거 후 기존 단일 함수 구조로 복귀
- `boundary.py`에서 `_raise_matrix_size_error()` 제거 후 기존 직접 raise 구조로 복귀
- 필요 시 `tests/refactor/test_refactor_contract_safety.py` 제거로 테스트 보강분만 되돌리기 가능
